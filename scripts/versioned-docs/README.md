# Versioned Provider API Docs

This directory implements the **write-once archive** side of the registry's
versioned API docs, complementing the in-Hugo versioned docs generated by
`scripts/generate-versioned-docs.sh`. The design mirrors pulumi/docs'
versioned SDK/CLI docs system (`pulumi/docs:scripts/versioned-docs/README.md`).

## Architecture

Two tiers serve historical provider versions, unified behind one URL scheme
(`/registry/packages/<pkg>@<major>.x/…`) and one version-selector dropdown:

1. **In-Hugo (blessed first-party providers only).** On every site build,
   `scripts/generate-versioned-docs.sh` regenerates the last 3 majors (latest
   minor of each) from their pinned schemas into the normal Hugo content tree.
   These pages ship with each deploy and live in the per-deploy origin bucket.

2. **Archive (everything else).** All other packages' older majors — and
   blessed majors that age out of the in-Hugo window — are **immutable,
   write-once HTML snapshots** in a permanent S3 bucket
   (`pulumi-registry-versioned-{testing,production}`, owned by
   `infrastructure/versioned-docs/`). They are built once per (package,
   version), published with `Cache-Control: public, max-age=31536000,
immutable`, and never rebuilt.

Serving: the registry CloudFront distribution (`infrastructure/index.ts`,
gated on the `versionedDocsStack` stack config) adds

- an **origin group** on `/registry/packages/*@*`: primary = the per-deploy
  origin bucket, failover on 403/404 = the archive bucket. An in-Hugo `@` page
  is served from the deploy; when it ages out of the window, the same URL
  transparently fails over to the archive. **No URL ever changes.**
- a direct archive behavior on `/registry/versioned/*` for the per-package
  manifests.

```
/registry/packages/aws@6.x/...     in-Hugo today; archive after the window rolls
/registry/packages/navs/aws@6.x.json   nav tree (same failover)
/registry/versioned/aws/versions.json  manifest (archive only, max-age=300)
```

## The manifest

One JSON per package at `registry/versioned/<pkg>/versions.json`
(`Cache-Control: public, max-age=300`), listing ALL versions so the dropdown
is unified:

```json
{
    "package": "aws",
    "liveRoot": "/registry/packages/aws/",
    "versions": [
        {
            "version": "v7.10.0",
            "slug": "",
            "path": "/registry/packages/aws/",
            "latest": true,
            "source": "hugo"
        },
        {
            "version": "v6.83.0",
            "slug": "6.x",
            "path": "/registry/packages/aws@6.x/",
            "source": "hugo"
        },
        {
            "version": "v4.38.1",
            "slug": "4.x",
            "path": "/registry/packages/aws@4.x/",
            "source": "archive",
            "date": "…"
        }
    ]
}
```

`source` is operational metadata; thanks to the shared URL space the selector
doesn't care where a version is served from. `source: "hugo"` entries are
refreshed by `reconcile-package.sh` on every version bump; `source: "archive"`
entries are appended by `publish-version.sh`.

## The selector

- **Contract URL (permanent, never move it):** `/registry/js/versioned-docs.js`
  (`themes/default/static/registry/js/versioned-docs.js`, plus its CSS
  sibling). It is served by the live site, so archive chrome stays current
  without republishing snapshots.
- **Live pages** get the loader tag at Hugo build time
  (`partials/registry/package/package-card-top-of-page.html`,
  `data-vdocs-mode="latest"`). The loader fetches the manifest and merges
  archived versions into the build-time `<select>` (baked in-Hugo entries win),
  or mounts a dropdown when the build rendered none (single-version packages).
- **Archive pages** get their tag injected at publish time
  (`inject-version-switcher.sh`, `data-vdocs-mode="archive"`), along with
  `<meta name="robots" content="noindex">` and a `<link rel="canonical">`
  pointing at the live-equivalent page. The loader replaces the frozen baked
  dropdown wholesale from the manifest.
- The loader is **fail-silent**: on manifest 404 (PR previews, packages with no
  archive) pages behave exactly as before.

## Snapshot self-containment

Registry API pages functionally require the site JS (the left nav and language
chooser are Stencil web components), and both JS and CSS bundles are
content-hashed in per-deploy buckets that get deleted. `build-snapshot.sh`
therefore **vendors** every deploy-fingerprinted asset into `_vassets/` inside
the snapshot prefix and rewrites references: `/registry/js/*`,
`/registry/css/*`, the `/icons/*.svg` sprite, and `/fingerprinted/*` logos.
Stable committed paths (`/fonts/*`, `/images/*`, `/logos/*`) stay absolute.

## Scripts

| Script                       | Purpose                                                                                                                                                                                                                                                   |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `lib.sh`                     | Shared per-(package, version) generation core (schema download/cache, resourcedocsgen invocation, nav/schema/data emission). Sourced by `generate-versioned-docs.sh` and `build-snapshot.sh`.                                                             |
| `discover-versions.sh <pkg>` | Version + schema-URL discovery. Blessed packages → private `registry-mirror-discover`; everything else → `resourcedocsgen versions` (GitHub tags + schema_file_url substitution + Registry API fallback; unresolvable versions land in `skipped`).        |
| `build-snapshot.sh`          | Builds one publishable snapshot: generate content → targeted Hugo build → extract → vendor assets. `--out-dir` output is previewable locally.                                                                                                             |
| `inject-version-switcher.sh` | Publish-time HTML post-processor: noindex + canonical→latest + loader. Idempotent.                                                                                                                                                                        |
| `publish-version.sh`         | Write-once publish: refuse-overwrite (unless `--force`), inject on a copy, immutable S3 sync (+ nav JSON + `.vdocs.json` metadata), manifest read-modify-write, CloudFront invalidation. `--manifest-only --hugo-window` refreshes just the hugo entries. |
| `reconcile-package.sh <pkg>` | The policy brain: discovery → desired hugo/archive split → bucket diff → plan JSON (`archive[]` with `published` flags). `--update-manifest` refreshes the manifest's hugo window.                                                                        |
| `rebuild-manifest.sh`        | Race-proof manifest reconciliation from the bucket listing (uses each snapshot's `.vdocs.json`). Run after parallel backfills.                                                                                                                            |
| `redact-version.sh`          | Admin-only: permanently delete a snapshot (all S3 object versions) + drop it from the manifest. The CI publisher role deliberately cannot do this.                                                                                                        |
| `assert-head-tag.sh`         | CI guard: every snapshot page must have an injection point.                                                                                                                                                                                               |

## Workflows

| Workflow                      | Trigger                                                                  | Does                                                                                                                                                                                                        |
| ----------------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `archive-package-version.yml` | `workflow_dispatch` / `workflow_call`                                    | Build + publish ONE (package, version) snapshot. Self-activating: no-ops when the archive stack isn't stood up.                                                                                             |
| `archive-backfill.yml`        | `workflow_dispatch`                                                      | Plan across all (or filtered) packages via `reconcile-package.sh`, then publish the worklist with bounded parallelism. Refuse-overwrite makes reruns idempotent/resumable; skips are reported, never fatal. |
| `archive-reconcile.yml`       | push to `master` touching `themes/default/data/registry/packages/*.yaml` | For each bumped package: refresh the manifest's hugo window and archive any major that left the retention window. Failures never block deploys; state self-heals on the next bump.                          |

## Retention policy

Last **3 majors** per package (latest minor of each):

- Blessed: 3 majors in-Hugo; the archive receives a major when it leaves the
  window (plus one reach-back major at backfill time).
- Non-blessed: latest in-Hugo; 2 older majors archived at backfill.
- Archives are write-once and **accumulate** — a version, once archived, stays
  available forever (until deliberately redacted).
- Versions whose historical schemas can't be fetched are skipped and reported
  (best-effort by design).

## Runbooks

**Enable the system in an environment**

1. `cd infrastructure/versioned-docs && pulumi up -s pulumi/pulumi-registry-versioned/<env>`
2. Set `registry:versionedDocsStack: pulumi/pulumi-registry-versioned/<env>` in
   `infrastructure/Pulumi.<env>.yaml` and let the next deploy run `pulumi up`
   (adds the origin group + behaviors; purely additive — verify with
   `pulumi preview`).
3. Publishing self-activates: workflows resolve the bucket via
   `pulumi stack output`; while the stack doesn't exist they no-op.

**Publish one version by hand**

```bash
gh workflow run archive-package-version.yml \
  -f package=aiven -f version=v6.30.0 -f target_env=testing
```

**Backfill** — run `archive-backfill.yml` (optionally with `package_filter`,
`dry_run=true` first). Rerunning is safe: published versions no-op. After a
large parallel backfill, run `rebuild-manifest.sh` per package if manifest
races are suspected.

**Local dry run (no AWS)**

```bash
./scripts/versioned-docs/build-snapshot.sh --package aiven --version v6.30.0 \
  --schema-url https://raw.githubusercontent.com/pulumi/pulumi-aiven/v6.30.0/provider/cmd/pulumi-resource-aiven/schema.json \
  --out-dir /tmp/aiven-snap
python3 -m http.server -d /tmp/aiven-snap/snapshot
```

**Redact a version** — `redact-version.sh` with admin credentials (see script
header).

## Out of scope / follow-ups

- **LLM docs**: archived versions do not emit `llm-docs.json`; only the latest
  version's bundle is published. Wiring `--cliDocsOutDir` into the versioned
  pipeline is a follow-up.
- The old manual snapshot flow (`docs/adding-previous-version-snapshot.md`,
  `aws-v6`-style names) is superseded; `infrastructure/redirects.js` redirects
  those legacy URLs to the `@` scheme.
